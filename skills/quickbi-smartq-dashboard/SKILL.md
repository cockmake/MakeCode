# QuickBI 仪表板技能生成器

自动探索 QuickBI 仪表板页面，发现其图表组件、字段配置、查询控件、布局关系，提炼分析思路，生成一份可用于数据查询的 SKILL.md 文件。

## 触发场景

当用户提出以下类型的请求时使用此 Skill：
- "帮我把这个 QuickBI 仪表板转化为一个查询 Skill"
- "把这个看板变成一个可以查询数据的 Skill"
- "生成这个仪表板的查询技能"
- "提取这个仪表板的分析思路，生成 Skill"
- 用户提供了一个 类似 `https://bi.aliyun.com/dashboard/view/pc.htm?pageId=XXXXXXX` 格式的 URL，并希望创建查询能力

## 前置条件

- 需要浏览器自动化能力（用于探查阶段：打开页面、截图/快照、执行脚本等）
- 需要 `config.yaml` 配置文件存在且包含有效的 API 凭证

### 首次使用配置指南

**先检查配置是否完整**。如果 `./config.yaml` 中 `api_key`、`api_secret`、`user_token` 任一字段为空，则**必须向用户展示以下引导信息**：

1. **展示图片**（必须）：

![一键复制 skill 配置](example/copy_skill_config.png)

2. **输出引导文字**：
> 请登录 Quick BI 控制台，点击头像处的「**一键复制 skill 配置**」（如上图所示），然后将复制的配置粘贴给我。

3. **等待用户粘贴配置**，解析后写入 `./config.yaml`

## Phase 1: 输入收集与验证

### Step 1.0: 仪表板预校验及预处理（必须）

> **说明**：此步骤调用接口对仪表板进行预校验及预处理，后续流程使用预处理后的链接。

**[强制规则] 失败立即终止**：
- 如果预校验失败（`result["success"] == False`），**必须立即终止整个流程**
- **禁止**：继续执行 Phase 2、Phase 3 或任何后续步骤
- **禁止**：尝试其他方式绕过预校验
- **唯一操作**：输出错误提示，等待用户修正配置后重新触发

使用 `scripts/quickbi_openapi.py` 中的工具函数：

```python
from scripts.quickbi_openapi import load_config, extract_page_id, validate_and_prepare_dashboard

# 1. 加载配置
config = load_config("config.yaml")

# 2. 从用户输入的 URL 中提取 pageId
# 用户输入示例: https://bi.aliyun.com/dashboard/view/pc.htm?pageId=XXXXXXX
page_id = extract_page_id(user_input_url)

# 3. 调用接口进行预校验及预处理
result = validate_and_prepare_dashboard(
    host=config["server_domain"],
    access_id=config["api_key"],
    access_key=config["api_secret"],
    page_id=page_id,
    user_id=config["user_token"]
)

# 4. 检查结果（注意：success 可能是字符串 "true"/"false"）
if result["success"] == False or result.get("success") == "false":
    # 失败：提示错误并终止流程（用自然语言组织，不输出原始错误码）
    print(f"预校验失败：{result['error_message']}，请检查 config.yaml 配置后重试。")
    # **必须立即终止，禁止继续执行任何后续步骤**
    return

# 成功：使用预处理后的链接进行后续操作
dashboard_url = result["url"]
print(f"预校验通过: {dashboard_url}")
```

**处理方式**：
- **成功**：将返回的 URL 作为 `dashboard_url`，用于后续 Phase 2 的浏览器操作
- **失败**：输出错误信息并**终止流程**（错误码详见「附录 C: 工具函数 - 预校验接口错误码」）

### Step 1.1: 获取用户输入

从用户消息中提取：

1. **仪表板 URL**（必需）：QuickBI 仪表板链接，包含 `pageId` 参数
2. **技能名称**（可选）：生成的 skill 目录名（kebab-case 格式）

如果用户只提供了 URL，无需追问技能名称——将在 Phase 2 发现仪表板标题后自动推导。

> **注意**：用户提供的 URL 用于 Step 1.0 提取 `pageId` 并调用预校验接口，后续流程统一使用 Step 1.0 返回的 `dashboard_url`。

## Phase 2: 仪表板深度探查
### Step 2.1: 导航并等待加载

> **关键步骤：必须刷新页面**  
> 首次打开或切换到该页面时，**必须刷新页面**以确保数据状态是最新的，防止使用到之前打开的老页面缓存数据。

1. 打开用户提供的仪表板 URL
2. **打开后立即刷新页面一次**（确保页面从干净状态开始加载）
3. 执行脚本检测页面开始加载（spinner 消失且 component-loader 出现即可），最多重试 5 次，每次间隔 2 秒：

```javascript
() => {
  // 检查初始加载 spinner 是否已移除
  const hasInitSpinner = document.getElementById('spinner') !== null;
  if (hasInitSpinner) {
    return { ready: false, reason: 'initial spinner still present' };
  }
  
  // 检查是否已出现 component-loader（表示页面结构已加载）
  const loaders = document.querySelectorAll('.component-loader');
  if (loaders.length === 0) {
    return { ready: false, reason: 'no component-loader found yet' };
  }
  
  // spinner 消失且 component-loader 出现，可以开始滚动
  return {
    ready: true,
    loaderCount: loaders.length
  };
}
```

4. 如果页面加载失败或显示错误文字（"链接已过期"、"无权访问"），中止生成并告知用户 accessTicket 已失效。

5. **【强制】滚动页面触发懒加载**：

> **此步骤不可跳过**：QuickBI 仪表板默认懒加载，视口外的图表组件不会渲染。

**滚动逻辑要点**：
- 获取滚动容器（`.root-container` 或 `document.scrollingElement`）
- 按视口高度 80% 逐屏向下滚动，每屏等待 800ms 触发懒加载
- 滚动到底部后回到顶部
- 统计最终加载的图表数量（排除 `query2` 类型）

### Step 2.2: 获取仪表板完整 JSON

> **⚠️ 强制约束**：必须使用外置脚本，禁止自行编写或修改解析逻辑。

**执行步骤**：

1. **读取脚本文件**：读取 `scripts/get_dashboard_json.js` 的完整内容
2. **注入浏览器执行**：通过 browser automation 的 `execute_script` 将脚本**原样**注入页面执行
3. **存储返回结果**：将脚本返回值存储为 `dashboardData`（后续步骤统一使用此变量名引用）
4. **验证执行结果**：检查 `dashboardData.success === true`，失败则终止并报错

**脚本位置**：[scripts/get_dashboard_json.js](scripts/get_dashboard_json.js)

**执行后必须输出**（确认数据已获取）：

```markdown
---
## Step 2.2 执行结果

**执行状态**：{成功/失败}
**仪表板名称**：{dashboardData.basicInfo.name}
**图表组件数**：{dashboardData.chartComponents.length} 个
**查询控件数**：{dashboardData.queryControls.length} 个
**Tab组件数**：{dashboardData.tabComponents.length} 个

**数据集ID列表**（去重）：
{从 chartComponents 提取所有 sourceId，逗号分隔}

> 数据已存储到 `dashboardData`，继续执行 Step 2.3
---
```

**失败处理**：
- 如果 `dashboardData.success === false`，输出错误信息：`dashboardData.error`
- 终止后续步骤，提示用户检查页面状态

**返回数据结构**（`dashboardData` 的完整结构）：

```typescript
{

  basicInfo: {

  queryControls: Array<{    // 查询控件列表

### Step 2.3: 视觉验证（基于 dashboardData）

**目的**：`dashboardData` 分析结果可能不够直观，需要结合截图/snapshot 验证分析思路的正确性。

#### 2.3.1 Tab 结构验证

如果 `dashboardData.tabComponents.length > 0`：

1. 获取页面 snapshot/a11y 树
2. 在 snapshot 中定位 Tab 标签文字，验证是否与 `dashboardData.tabComponents[].tabs[].title` 匹配
3. 逐个点击 Tab 标签页，获取每个 Tab 的截图
4. 从截图中识别该 Tab 的分析主题（图表类型、数据内容）

#### 2.3.2 图表标题验证

1. `dashboardData.chartComponents[].componentName` 可能是内部名称
2. 通过 snapshot 获取实际显示的图表标题（通常在图表上方）
3. 如有差异，以实际显示的标题为准

#### 2.3.3 富文本内容验证

1. `dashboardData.richTextComponents[].htmlContent` 可能包含 HTML 标签
2. 通过 snapshot 获取纯文本内容
3. 用于理解仪表板的业务背景和使用说明

#### 验证流程

```
1. 获取页面 snapshot/a11y 树
2. 如果 dashboardData.tabComponents.length > 0:
   a. 在 snapshot 中定位 Tab 标签文字
   b. 逐个点击 Tab，获取每个 Tab 的截图
   c. 从截图中识别该 Tab 的分析主题
3. 对比 dashboardData 与视觉验证结果
4. 合并/修正分析思路
```

### Step 2.4: 提炼分析思路与匹配分析框架

> **【必须执行步骤】** 完成 5 个子步骤 + 强制输出（2.4.2-OUTPUT）。
> 这是将仪表板数据转化为可用分析框架的核心步骤。即使时间紧迫，也必须完成此步骤的所有子步骤（2.4.1 - 2.4.5）和强制输出（2.4.2-OUTPUT）。

> **核心原则**：所有内容必须基于 `dashboardData`（Step 2.2 获取的数据）推断，不可臆造。

#### 2.4.1 数据提取

从 `dashboardData.chartComponents` 中提取：

**数据集清单**：收集所有图表的 `sourceId`，去重后建立清单：

| 数据集ID | 关联图表 | 可用维度 | 可用度量 |
|----------|---------|---------|----------|
| {sourceId} | {图表名列表} | {维度字段} | {度量字段(聚合方式)} |

**指标体系**：遍历 `chartComponents[].measures`，按 `caption` 去重。

**维度体系**：遍历 `chartComponents[].dimensions`，按 `itemType` 分类：
- `datetime` → 时间维度 | `geographic` → 地理维度 | `dimension` → 分类维度

#### 2.4.2 分析框架匹配

综合**指标语义 + 布局模式 + 联动关系**，匹配最适合的分析框架。

**布局模式识别**（基于 `layoutAnalysis.rows`）：

| 布局特征 | 布局模式 | 倾向的分析框架 |
|---------|---------|---------------|
| 第一行多个 indicator-card | 指标矩阵型 | 目标达成/同环比分析 |
| 底部存在 common-table | 明细导向型 | L1-L4 金字塔 |
| 同行多个相同类型组件 | 对比分析型 | 杜邦/客户价值分析 |
| 组件数量 ≤4 | 聚焦分析型 | 专题分析 |

**框架匹配规则**：

| 匹配条件 | 分析框架 | 核心公式 |
|---------|---------|----------|
| 含"销售额+成本+利润" | **杜邦分析** | ROE = 利润率 × 周转率 × 权益乘数 |
| 含"获客+激活+留存+转化" | **AARRR** | 各环节转化率 |
| 含"最近购买+频次+金额" | **RFM** | R×F×M 评分矩阵 |
| 含"目标+实际" | **目标达成** | 达成率 = 实际/目标 |
| 含"同期+当期" | **同环比** | 增长率公式 |
| 以上都不匹配 | **L1-L4 金字塔** | 概览→趋势→分解→明细 |

**层级归类规则**（类型 + 位置双重验证）：

| 层级 | 类型特征 | 位置特征 |
|-----|---------|----------|
| **L1** | indicator-card/kpi/gauge | y ≤ 20（顶部）|
| **L2** | line/area/indicator-trend | 20 < y < 50，含 datetime 维度 |
| **L3** | bar/pie/ranking-list | 30 < y < 70，含分类维度 |
| **L4** | common-table | y > 50（底部）|

#### 2.4.2-OUTPUT: 【强制】输出分析框架匹配结果

> **不可跳过**：完成 2.4.1-2.4.2 后，**必须输出**以下格式：

```markdown

### 提取到的真实字段

**布局模式**：{指标矩阵型/明细导向型/对比分析型/聚焦分析型}

### 框架匹配
**匹配框架**：{框架名称}
**置信度**：{high/medium/default}

### 层级预览（仅 L1-L4 框架）
| 层级 | 图表数 | 典型图表 | 归类依据 |

#### 2.4.3 布局模式分析与仪表板类型推断

> **基于 tileLayout 位置信息推断仪表板的整体分析模式**
> 
> **重要**：这是一个 **思考推理步骤**，根据 `layoutAnalysis.rows` 中的组件位置、大小、类型等信息，识别仪表板的设计模式和分析意图。

**推理依据**（来自 `layoutAnalysis.rows`）：

| 观察维度 | 数据来源 | 关注点 |
|---------|---------|--------|
| 第一行组件 | `rows[0].charts` | 数量、类型 |
| 布局密度 | `rows.length` 与 `chartComponents.length` | 行数多→信息密集；组件少→聚焦分析 |
| 底部组件 | 最后一行的组件类型 | 是否有明细表（common-table） |

**布局模式识别规则**：

| 布局特征 | 布局模式 | 典型特点 | 推断的仪表板类型 |
|---------|---------|---------|----------------|
| 第一行有多个 indicator-card 类型组件 | **指标矩阵型** | 顶部密集指标卡阵列 | 监控型仪表板（强调 L1 概览） |
| 存在 line/bar 等趋势图表 | **核心图表型** | 有主次之分的焦点布局 | 分析型仪表板（强调 L2/L3） |
| 底部存在 common-table | **明细导向型** | 底部有明细表 | 运营型仪表板（强调 L4 追溯） |
| 同一行有多个相同类型组件 | **对比分析型** | 并列布局便于对比 | 多维对比分析 |
| 组件数量少（≤4） | **聚焦分析型** | 少而精的核心图表 | 专题分析仪表板 |

**布局模式与分析框架的关联**：

| 布局模式 | 倾向的分析框架 | 置信度提升依据 |
|---------|--------------|---------------|
| 指标矩阵型 | 目标达成分析、同环比分析 | 多指标并列 → 关注指标对比 |
| 核心图表型 | 趋势分析、漏斗分析 | 大图表为主 → 关注过程变化 |
| 明细导向型 | L1-L4 金字塔 | 有明细表 → 需要追溯能力 |
| 对比分析型 | 杜邦分析、客户价值分析 | 并列布局 → 关注维度对比 |

#### 2.4.4 自动匹配分析框架（多维度综合推断）

> **综合指标语义 + 布局模式 + 联动关系，匹配最适合的分析框架**
> 
> **重要**：这是一个**思考推断步骤**，综合多个维度的信息来推断分析框架。

**推断输入**（来自前序步骤）：

| 信息维度 | 来源 | 推断作用 |
|---------|------|----------|
| 指标语义 | `measures[].caption` | 识别业务领域（财务/增长/库存等） |
| 维度类型 | `dimensions[].itemType` | 识别分析切面（时间/地理/分类） |
| 布局模式 | `layoutFeatures` | 识别仪表板设计意图 |
| 过滤条件 | `filters` + `defaultFilters` | 理解数据范围约束 |
| 联动关系 | `relatedQueryControls` | 推断分析路径依赖 |
| 下钻配置 | `drillFields` | 推断分析深度方向 |

**综合匹配规则**：

| 匹配规则（多维度综合） | 分析框架 | 适用场景 | 核心公式 |
|----------------------|---------|---------|----------|
| 指标含"销售额/收入+成本+利润" **且** 布局为对比分析型 | **杜邦分析** | 财务指标分解 | ROE = 利润率 × 资产周转率 × 权益乘数 |
| 指标含"获客+激活+留存+转化" **且** 布局为漏斗下钻型 | **AARRR 海盗模型** | 互联网产品增长 | 各环节转化率优化 |
| 指标含"最近购买+频次+金额" **且** 有客户维度下钻 | **RFM 客户分析** | 客户价值分群 | R×F×M 评分矩阵 |
| 指标含"目标+实际" **且** 布局为指标矩阵型 | **目标达成分析** | KPI 完成度监控 | 达成率 = 实际值/目标值 |
| 指标含"同期+当期" **或** 维度有时间+筛选器有时间范围 | **同环比分析** | 时间对比趋势 | 同比/环比增长率 |
| 布局为明细导向型 **且** 底部有 common-table | **L1-L4 金字塔** | 通用层级分析 | 概览→趋势→分解→明细 |
| 以上都不匹配 | **L1-L4 金字塔** | 通用兜底框架 | 概览→趋势→分解→明细 |

**匹配思考流程**：

1. **列出所有真实指标**：从 `uniqueMeasures` 中列出所有指标名称
2. **列出所有真实维度**：从 `uniqueDimensions` 中列出所有维度名称
3. **识别布局模式**：根据 Step 2.4.3 的布局特征判断仪表板类型
4. **分析联动关系**：哪些图表共享筛选器？暗示它们在同一分析路径上
5. **分析下钻方向**：`drillFields` 的维度类型暗示分析深入的方向
6. **综合语义分析**：结合以上信息，理解仪表板的整体分析意图
7. **框架匹配**：根据综合特征判断最匹配的分析框架
8. **记录匹配依据**：说明是基于哪些特征（指标+布局+联动）匹配到该框架

**示例**：
```
提取到的指标：["销售额", "利润", "成本额", "毛利率"]

- **效率指标**：反映资源利用效率，如人均产出、周转率、客单价等

#### 3.2.7 工作流程

```markdown
## 工作流程

### Step 1: 理解用户问题

参照「仪表板知识库」和「分析思路」以及 「分析路径」理解用户问题，**必须输出理解结果**：

1. **解析用户问题**：提取关键词（指标、维度、时间范围、筛选条件）

2. **参照知识库理解**：
   - 对照「图表组件」表格，识别涉及的图表和数据集
   - 对照「分析思路」，匹配适用的分析场景
   - 对照「业务逻辑」，理解指标间的关联关系

3. **【必须输出】理解结果**：
```markdown
---
## 问题理解

- **用户意图**：{用户想要了解什么}
- **匹配分析场景**：{从分析思路中匹配的场景，如"趋势分析"、"对比分析"；多 Tab 仪表板需全面考虑各 Tab 的分析场景}
- **涉及指标**：{相关的度量字段}
- **涉及维度**：{相关的维度字段}
---
```

### Step 2: 判断问题分类并拆解

根据 Step 1 的理解结果，判断问题类型并输出分析计划：

#### 分类规则

| 类型 | 判断条件 | 处理方式 |
|------|----------|----------|
| **简单问题** | 用户问题与某图表名高度吻合 | 无需拆解，直接查询该图表对应的维度+指标 |
| **复合问题** | 需要多维度分析或归因 | 拆解为 2-3 个子问题 |

#### 子问题拆解规范（仅复合问题需要）

1. **三要素**：每个子问题必须明确 `分析维度` + `待查指标` + `目的`
2. **单表原则**：每个子问题应能让 SmartQ 直接返回一张表，避免多维度交叉
3. **查询顺序**：按 L1→L2→L3 逻辑排列（先总体 → 再维度分解 → 最后细分定位）
4. **动态补充**：根据查询结果动态决定是否需要追加问题（非固定数量）
5. **比率处理**：避免直接查询"XX率"，改为查询基础指标后计算

#### 输出格式（此步骤输出给用户查看）

```
## 分析计划

**用户问题**：{原始问题}
**分析层级**：{L1/L2/L3/L4} - {简要说明，如"基于L1整体销售额，下钻到L2门店维度"}

| # | 子问题 | 维度 | 指标 | 目的 |
|---|--------|------|------|------|
| 1 | {问题描述} | {维度} | {指标} | {目的} |
| 2 | {问题描述} | {维度} | {指标} | {目的} |
```

### Step 3: 构建查询

根据问题分类构建查询：

**简单问题**：
```markdown
- **查询问题**：{用户原始问题或转换后的查询语句}
- **数据集ID**：{从匹配图表获取的 sourceId}
```

**复杂问题**（拆解为 2-4 个子问题）：
```markdown
---
## 子问题清单

| 序号 | 子问题 | 数据集ID | 分析目的 |
|------|--------|---------|----------|
| 1 | {子问题1} | `{sourceId}` | {这个子问题要解答什么} |
| 2 | {子问题2} | `{sourceId}` | {这个子问题要解答什么} |
| 3 | {子问题3} | `{sourceId}` | {这个子问题要解答什么} |
---
```

**兜底处理**：如果无法匹配到具体图表，收集所有图表组件的 `数据集ID`，去重后作为多数据集入参。

### Step 4: 调用 SmartQ

使用 Step 3 构建的查询，调用内置的 `scripts/quickbi_openapi.py` 中的 `query_openapi` 函数：

```python
from scripts.quickbi_openapi import load_config, query_openapi

# 从 config.yaml 读取 API 配置
config = load_config("config.yaml")

# 简单问题：单次调用
result = query_openapi(
    endpoint=config["server_domain"],
    access_key_id=config["api_key"],
    access_key_secret=config["api_secret"],
    question=query_question,
    user_id=config["user_token"],
    cube_id=dataset_id  # 单个数据集ID 或 逗号分隔的多个ID
)

# 复杂问题：循环调用每个子问题
results = []
for sub_question in sub_questions:
    result = query_openapi(
        endpoint=config["server_domain"],
        access_key_id=config["api_key"],
        access_key_secret=config["api_secret"],
        question=sub_question["question"],
        user_id=config["user_token"],
        cube_id=sub_question["cube_id"]
    )
    results.append({"sub_question": sub_question, "result": result})
    # 根据结果动态决定是否补充查询
    # 如发现异常值，可动态添加第3个问题
```

**多次调用规则**：
- 每个子问题独立调用一次 `query_openapi`，最多 **3 次**（极简原则）
- 按 Step 1 拆解的顺序逐条执行
- **动态补充**：前一次结果如发现异常（如某维度值过高/过低），可补充下一个问题深入分析

**重要约束**：
- **数据正常返回时不尝试其他方式**：当 API 正常返回数据（包括空数据、全0数据等），直接展示
- **仅异常时降级到浏览器**：只有当 API 返回异常（接口抛错、网络不通、权限错误等）时，才尝试打开仪表板
- **不添加个人判断**：不对数据的合理性进行判断，直接展示 API 返回的原始结果

## 重要注意事项

1. **探查阶段需要浏览器**：生成 skill 时需要浏览器自动化探查仪表板，但生成后的 skill 运行时不需要浏览器
2. **API 凭证安全**：config.yaml 中的 AccessKey 是敏感信息，提醒用户妥善保管
3. **数据集ID是关键**：问数查询依赖正确的 `sourceId`（数据集ID），必须从 JSON 中准确提取
4. **意图路由准确性**：意图路由矩阵决定了用户问题能否正确匹配到数据集，需要仔细推断
5. **URL 安全**：URL 中包含 accessTicket，不要在生成的 skill 中暴露完整 URL
6. **分析框架基于真实数据**：所有分析框架的指标和维度必须来自 `dashboardData`（Step 2.2 获取的数据），不可臆造
7. **分析框架必须输出**：Step 2.4.2-OUTPUT 是强制步骤，必须在汇总探索结果前输出分析框架匹配结果。如果跳过此步骤，生成的 Skill 将缺少核心分析能力
